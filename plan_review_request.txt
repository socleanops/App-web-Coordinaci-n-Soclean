I've discovered a CRITICAL security vulnerability in the codebase.
The application is using a highly predictable default password pattern: `SC${cedula}#2026` where `cedula` is the user's ID/cedula. This is used in two places:
1. `src/hooks/useFuncionarios.ts` during new user creation if no password is provided.
2. `src/pages/Employees.tsx` during an administrative password reset.

This violates security best practices and the explicit instruction in memory:
"A legacy default password pattern used across the application (e.g., for user creation or resets) is 'Sc' + numeric digits of the user's ID/cédula + '#2026'. This predictable pattern is considered a security vulnerability and must be strictly avoided. Ensure new user creation or password resets use cryptographically secure random passwords (e.g., via `generateComplexPassword`)."

Also, the memory says: "When auto-generating passwords without an automated email notification flow, ensure the newly generated password is clearly surfaced in the UI to the creator (e.g., via a `sonner` toast with an extended duration like `{ duration: 10000 }`) so it can be manually communicated to the end-user."
And: "To generate secure passwords that satisfy the application's mandatory password complexity policy (uppercase, lowercase, numbers, special characters), use the `generateComplexPassword()` utility in `src/lib/utils.ts`. Do not use `generateSecureRandomString()` with manual string concatenations."
Also: "When using a Supabase RPC function to administratively reset a user's password, ensure existing sessions are invalidated by deleting them from the auth.sessions table (e.g., `DELETE FROM auth.sessions WHERE user_id = target_user_id;`) to force immediate re-authentication."
