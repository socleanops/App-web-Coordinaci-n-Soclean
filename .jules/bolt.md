## 2024-05-18 - Hoisting Intl Objects and Timezone Safety
**Learning:** Instantiating `Intl.DateTimeFormat` inside tight loops (like iterating over 15-minute intervals) introduces severe performance bottlenecks (e.g. 1s vs 16ms for 10K iterations). However, attempting to replace it with native Date methods like `.getHours()` causes critical regressions due to local vs explicit timezone differences across deployment/testing environments. Furthermore, Vitest mocks using local time format strings (e.g., '2026-03-03T20:00:00') will behave differently based on the machine running them unless explicit timezone offsets ('-03:00') are provided.
**Action:** Always hoist `Intl.DateTimeFormat` configurations to the module scope to reuse the instance efficiently without compromising timezone correctness. Always enforce explicit offsets (`-03:00`) in mock test strings interacting with explicit timezone formatters to maintain green pipelines across environments.
